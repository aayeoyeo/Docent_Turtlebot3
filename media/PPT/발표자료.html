프로젝트 기획 및 핵심 요구사항

프로젝트 목표

ROS 2 환경의 터틀봇3와 스마트폰 센서 인프라를 웹소켓으로 융합하여, 전시관 내에서 관람객을 안내하고 작품을 해설하는 자율주행 AI 도슨트 시스템 구축

  • 웹 기반 통합 제어: 별도 앱 설치 없이 모바일 브라우저로 로봇 조작 및 해설 수신
  • 비동기 제어 흐름: 해설 시작(/tts_text)과 완료(/tts_done)를 '요청-완료 페어'로 묶어 제어

주요 요구사항 (MoSCoW)

  • Must 전시물 순회 (M1): 최소 3점 전시물 코스 순회 (QR 정지 오차 ±20cm)
  • Must 해설 재생 (M2): JSON 기반 20초 이상 한국어 TTS 해설 재생
  • Must 안전장치 (M7): 웹앱 백그라운드 전환 감지 등 안전장치 3종 적용
  • Should 관람 통계 (S3): 작품별 실제 로봇 체류 시간 산출 및 관제 기록

토픽 인터페이스 명세 (최종)

웹 클라이언트와 ROS 노드 간 비동기 흐름 제어를 위한 핵심 Topic 설계

주요 Publisher (Web -> ROS)
  • /course (String)

    관제탑에서 편집된 전시물 ID, 타이틀, 대본을 포함한 JSON 주입

  • /exhibit_seen (String)

    스마트폰 카메라가 QR 인식 시 디코딩된 전시물 ID(E1 등) 발행

  • /tts_done (Bool)

    Web Speech API 해설 재생 완료 플래그 (비동기 흐름 제어용)

주요 Subscriber (ROS -> Web)
  • /tour_state (String)

    현재 Phase, 인덱스 등 진행 현황을 대시보드에 실시간 동기화

  • /tts_text (String)

    목적지 도착 시 스마트폰으로 송출할 도슨트 해설 순수 텍스트

핵심 구현 1: 스마트폰 AI 파이프라인 (robot.html)

담당: 최준혁

모바일 통합 웹뷰 기반 제어

  • 비전 센싱: html5-qrcode 적용하여 모바일 환경 원시 QR 디코딩 (전시물 식별)
  • 얼굴 인식: tracking.js 라이브러리 포팅, 관람객 존재 여부를 판정하여 10Hz로 /audience 발행
  • 오디오 제어: Web Speech API를 통한 TTS 텍스트 재생. 재생 완료 시 /tts_done 이벤트를 발행하여 노드에 콜백 응답
  • UX 최적화: OS 오디오 세션 활성화를 위한 선제적 터치 유도 버튼 및 TTS 속도 조절(S2) UI 적용
요청-완료 비동기 제어 로직 (발췌)
// ROS로부터 해설 텍스트 수신 시 재생
ros.subscribe('/tts_text', function(msg) {
    let utterance = new SpeechSynthesisUtterance(msg.data);
    
    // 오디오 재생이 완전히 끝났을 때 이벤트 발생
    utterance.onend = function() {
        // ROS 노드에게 해설 종료 플래그 응답
        ros.publish('/tts_done', {data: true});
        startFaceTracking(); // 관람객 대기 상태(CHECK) 진입
    };
    
    window.speechSynthesis.speak(utterance);
});

def check_state_logic(self):
    current_time = time.time()
    
    # 1. 관람객이 확인된 경우 (정상 로직)
    if self.audience_detected:
        self.transition_to_next_move()
        
    # 2. 관람객 부재 정책 타임아웃 (M4)
    elif current_time - self.check_start_time > 20.0:
        self.get_logger().info('관람객 부재. 다음 코스로 강제 이동')
        self.transition_to_next_move()

메인 백엔드 제어 노드 구현

  • 상태 제어: MOVE ➔ EXPLAIN ➔ CHECK 기반 다단계 타이머 루프 구현
  • 경로 검증: 전시물 3점(E1➔E2➔E3) 연속 순회 E2E 자율 주행 성공 (QR 정지 오차 ±20cm 충족)
  • 부재 정책(M4): 해설 종료(CHECK 단계) 후 얼굴 미감지 상태가 20초간 지속 시 강제로 다음 목적지 이동 로직 완비
  • 관람 통계(S3): 각 전시물별 실제 체류 시간을 산출하여 배열로 기록

트러블슈팅 하이라이트

Day 3 ~ Day 4 통합 테스트 간 직면했던 예외 상황 대처 사례 (버그 번호: B-03)

증상 (Problem)

조명이 어둡거나 반사가 심한 구간 통과 시, 카메라 뷰어의 QR 코드 인식 실패로 인해 다음 단계로 진입하지 못하고 해당 구간에서 무한 주행하는 데드락 현상 발생

원인 (Cause)

상태 전이 트리거가 오직 /exhibit_seen 수신 여부에만 의존하여, 물리적 거리 대비 장시간 인식이 안 될 때를 대비한 가드 로직(Guard Logic) 부재

해결 (Solution)

구간별 이동 거리에 기반한 30초 타임아웃 정지 알고리즘 도입. 타이머 초과 시 강제로 주행을 멈추고 관제 대시보드에 에러 상태를 보고하도록 수정 (담당: 여태인)

기타 안정화 조치: 투어 진행 도중 관리자가 신규 /course를 실수로 재수신하더라도, 즉시 리셋하지 않고 현재 투어가 완전히 종료된 후 다음 사이클부터 반영되도록 큐(Queue) 정책 적용 완료.
